# 任务拆解

## 目标

把明确的方案设计拆成 AI agent 可以稳定执行的任务图。

本 skill 的重点不是重新设计方案，而是把方案变成可实施、可验证、可并发、可回滚的开发计划。拆解结果必须绝对遵循上游方案和用户边界，防止 AI 在实施时多改、少改、漏改、错改。

本 skill 是独立工具，可以单独用于需求实施拆解、技术方案落地拆解、bug 修复拆解、重构分阶段计划、多 agent 协作计划和提交前实施路线复核。它不依赖其他 skill；如有需要，只在最后建议后续交给 `implement-change`、`verify-change`、`review-code` 或 `commit-changes`。

## 默认模式

默认只读。

允许：

- 阅读用户给出的需求、方案、问题定位报告、上下文材料和相关代码。
- 使用只读命令理解项目规则、目录结构、调用链、测试命令、已有实现和 diff。
- 输出任务图、任务卡、依赖关系、并发批次、验证矩阵和实施风险。

禁止：

- 未经明确要求修改、创建、删除代码文件。
- 擅自改变上游方案、扩大范围或补充未经用户确认的新需求。
- 把尚未理解的方案强行拆成任务。
- 把“大块愿望”包装成可执行 task。
- 让多个 agent 并发修改同一文件、同一 lockfile、同一生成物或同一配置入口。

当用户明确要求“把任务计划写入文件/创建 issue/落地为文档”时，才允许进入写入模式；写入前仍要检查项目规则和 dirty worktree。

## 输入处理

优先识别输入来源：

- 用户直接提供的方案设计、需求说明、实施建议或架构决策。
- `research-approach` 输出的推荐方案和后续实现指导。
- `diagnose-problem` 输出的根因、最小修复建议和验证计划。
- 已有 issue、PRD、设计文档、diff、commit、日志或代码片段。

先提取并保留：

- 目标、非目标、验收标准和用户明确边界。
- 方案中的关键模块、接口、数据流、状态流和失败处理。
- 必须保持不变的行为、兼容性要求和核心链路。
- 已确认风险、待确认问题和上游方案中的假设。
- 用户是否要求单 agent 顺序执行、多 agent 并行、分阶段提交或只输出计划。

如果输入不足，不要直接拆任务。先补上下文或输出缺口。

## 理解门槛

拆任务前必须确认三件事。

### 1. 方案是否明确

至少明确：

- 要实现或修复什么。
- 明确不做什么。
- 成功标准是什么。
- 哪些约束来自用户，哪些来自项目，哪些只是推断。
- 是否涉及公共 API、IPC、schema、权限、数据、依赖、构建、发布或核心链路。

如果方案本身含糊、互相矛盾或缺少关键决策，先输出“方案缺口”和“最小确认问题”。不要替用户做高风险架构决策。

### 2. 项目上下文是否足够

至少理解：

- 适用的 `AGENTS.md`、README、贡献指南、测试说明和项目约定。
- 相关入口、调用链、数据流、状态归属、错误处理和测试位置。
- 已有相似实现、工具函数、服务、类型、mock、fixture 和配置。
- 当前 dirty worktree 状态，尤其是目标文件是否已有用户改动。
- 可用验证命令、验证成本和无法验证的路径。

如果没有足够项目上下文，先用只读方式补齐。低风险任务可以带着显式假设继续；中高风险任务不要输出最终任务图。

### 3. 改动范围是否可评估

至少评估：

- 会改哪些模块、文件、类型、接口、测试和配置。
- 哪些调用方会受影响。
- 哪些文件是共享冲突点。
- 哪些变更必须串行，哪些可以并行。
- 每一步完成后项目是否仍能构建、运行或通过局部验证。

如果改动范围不可评估，先输出“范围探针任务”，不要直接进入主体实现拆解。

## 拆解原则

### 最小稳定纵切

优先按可运行的行为纵切任务，而不是按纯技术层横切任务。

好的纵切 task 应该完成一个小而完整的行为片段，例如：

- 建立新入口的最小骨架，并保持旧路径不变。
- 接入一个调用方和对应测试，而不是一次性改所有调用方。
- 为一个错误分支补齐处理和验证，而不是泛泛“完善错误处理”。

只有在契约、类型、schema 或基础设施必须先稳定时，才使用横切前置任务。

### 契约优先

涉及公共契约时，先拆出契约确认或兼容层任务，再拆实现任务。

公共契约包括：

- API、IPC、事件、消息、CLI、配置、schema、数据库字段。
- 类型导出、barrel export、序列化格式、错误码。
- 插件 manifest、扩展协议、构建脚本、发布产物。

契约任务必须包含调用方检查、兼容策略和回滚方式。未经确认，不要安排破坏性契约变更。

### 探针先行

当上下文不足、风险未知或影响面不清时，先拆探针任务。

探针任务只用于收集证据，例如：

- 搜索调用方和动态入口。
- 验证测试命令是否可用。
- 建立最小复现或最小集成样例。
- 确认旧数据、旧版本或旧配置兼容路径。

探针任务不得顺手实现主体改动。

### 阶段化合并

高风险变更拆成稳定阶段：

1. 准备阶段：上下文、契约、测试基线、保护性日志或适配层。
2. 主体阶段：小纵切实现，每步保持项目稳定。
3. 接入阶段：逐个调用方或入口接入。
4. 验证阶段：回归、集成、E2E、手工 smoke、日志或截图证据。
5. 清理阶段：只在新路径稳定后删除兼容代码或旧入口；删除前必须有引用证明。

不要安排“先打破，再最后修好”的计划，除非用户明确接受隔离分支或独立 worktree。

## 任务粒度

一个合格 task 必须满足：

- 目标单一。
- 改动范围明确。
- 允许修改和禁止修改的文件清楚。
- 有足够上下文参考。
- 有可执行步骤。
- 有完成标准。
- 有验证方式。
- 有回滚或撤销边界。
- 不依赖未完成的隐含工作。
- 完成后项目处于稳定状态。

避免输出以下任务：

- “实现整个功能”。
- “调整所有调用方”。
- “补测试”。
- “重构这块”。
- “让联调通过”。
- “优化错误处理”。
- “处理边界情况”。
- “两个 agent 同时改同一个服务”。

如果任务仍然过大，继续拆到一个 agent 可以在单次上下文内完成并验证。

## 并发与多 agent

默认保守并发。只有满足以下条件才标为可并行：

- 文件所有权不重叠。
- 不共享 lockfile、生成物、全局配置、schema、barrel export 或入口注册文件。
- 不依赖同一个尚未稳定的契约。
- 验证命令互不污染状态。
- 合并顺序明确，冲突点有单独 owner。

多 agent 任务必须写清：

- 每个 agent 的任务边界。
- 允许读取范围。
- 允许修改范围。
- 禁止修改范围。
- 输入上下文。
- 输出格式。
- 汇总 agent 如何合并结果。
- 合并前必须重新检查 `git status`、相关 diff 和目标文件最新内容。

高风险并行实现优先建议使用独立 git worktree 或补丁交付，不要共享同一个 dirty worktree。

## 依赖关系

每个 task 标注依赖类型：

- `blocks`：必须完成后才能开始下游任务。
- `parallel`：可与同批次任务并发。
- `merge-gate`：必须合并并验证后才能进入下一阶段。
- `review-gate`：需要人工或 review skill 检查后才能继续。
- `confirm-gate`：涉及高风险决策，必须用户确认。

不要只给线性列表。即使最终顺序执行，也要说明为什么不能并行。

## 验证设计

每个 task 必须包含验证方式。验证从低到高选择：

1. 静态检查：类型、lint、导入导出、格式约束、schema 校验。
2. 单元测试：目标函数、边界、错误路径。
3. 集成测试：调用链、状态流、真实适配层。
4. E2E 或 smoke：用户路径、CLI、UI、服务启动。
5. 日志或可观测性验证：关键事件顺序、错误类别、降级路径。
6. 手工验证：只在自动化成本过高时使用，并写清步骤和预期。

验证计划必须覆盖：

- 原方案的验收标准。
- 旧行为不回归。
- 失败路径和边界条件。
- 核心链路稳定性。
- 多 agent 合并后的整体验证。

如果某个 task 无法验证，不能称为完成任务；必须改成探针任务、拆小任务或明确剩余风险。

## 质量门

输出前自检：

- 是否遵守只读边界。
- 是否没有擅自修改上游方案。
- 是否理解了方案设计和项目上下文。
- 是否识别了公共契约、核心链路、共享文件和高风险点。
- 是否先安排了必要的探针、契约或测试基线任务。
- 是否每个 task 都可执行、可验证、可回滚。
- 是否每个阶段后项目仍处于稳定状态。
- 是否给出了依赖关系、并发批次和合并门。
- 是否避免多个 agent 并发写同一文件或共享状态入口。
- 是否明确了不做范围，防止多改、少改、漏改、错改。
- 是否为后续实现 agent 提供了足够上下文参考和质量验证。

## 后续交接

- 需要开始编码：交给 `implement-change`，附带任务卡、允许修改范围和验证方式。
- 需要验证计划或结果：交给 `verify-change`，附带验证矩阵、命令和预期证据。
- 需要审查任务计划或合并结果：交给 `review-code`。
- 需要重新评估方案合理性：交给 `research-approach`，附带当前缺口和风险。
