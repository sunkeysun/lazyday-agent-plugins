# Git 提交

## 目标

基于当前工作区的真实改动，生成准确、符合项目规范的提交信息，并在用户明确要求时完成 `git commit` 和 `git push`。

本 skill 是独立工具，不依赖其他 skill。即使没有前置上下文，也必须自己完成工作区检查、改动理解、提交规范识别、提交范围确认和结果汇报。

## 模式判断

先判断用户要什么：

- 只生成提交信息：只分析，不执行 `git add`、`git commit`、`git push`。
- 创建提交：可以在确认提交范围后执行 `git add` 和 `git commit`。
- 提交并推送：可以在提交成功后执行 `git push`。
- 只问“怎么提交”“提交信息怎么写”：默认只读，只给建议。

除非用户明确要求提交或推送，否则不得改变 Git 状态。

如果用户说“提交上次那个需求”“提交当前需求”但无法从对话判断需求范围，先使用 `resume-context` 恢复候选需求；真正创建提交前仍必须基于当前 Git diff 确认提交范围。

## 产物交接

需求产物由 `manage-artifacts` 统一管理。需要创建、选择、切换、列出、归档或修复 `.lazyday/coding/requirements/` 时，先使用 `manage-artifacts`。如果用户说“继续”“上次”“当前需求”但没有明确需求 ID，先使用 `resume-context` 匹配候选需求。

当前需求已明确时，本 skill 只读取和更新与 Git 提交相关的产物。

本 skill 优先读取：

- `brief.md`：需求标题、目标和非目标，用于判断提交主题。
- `plan.md`：任务范围和文件所有权。
- `implementation.md`：实际改动摘要。
- `verification.md`：已运行验证和未覆盖风险。
- `review.md`：review 结论和阻塞问题。
- `handoff.md`：上一阶段最短交接摘要。

当用户要求跨会话记录、继续交接或允许落盘，且当前需求已明确时，把提交范围、提交信息、提交哈希和推送结果写入 `commit.md`，并同步更新 `handoff.md`。

`.lazyday/` 默认是本地协作产物，不纳入业务提交。除非用户明确要求提交协作产物，否则把 `.lazyday/` 归类为不建议纳入的无关产物。

如果用户明确限定“只使用本 skill”“只用 commit-changes”或限定本轮只能使用某几个 skill，不得自动转交 `resume-context`、`manage-artifacts` 或其他后续 skill；只能在输出中给出建议。

## 基线检查

开始时使用只读命令建立基线：

```bash
git rev-parse --show-toplevel
git status --short --branch
git log -n 20 --pretty=format:%s
```

再查找提交规范和项目上下文：

- 从仓库根和相关文件目录向上查找 `AGENTS.md`、`CLAUDE.md`、`CONTRIBUTING.md`、`README.md`。
- 检查 `.commitlintrc*`、`commitlint.config.*`、`package.json`、`.github/`、提交模板、PR 模板。
- 读取最近提交标题，识别是否使用 Conventional Commits、中文提交、工单号、TAPD/Jira/Linear 编号或固定前缀。
- 如分支名包含任务编号，结合项目历史判断是否应写入提交信息。

## 改动理解

提交信息必须来自真实 diff，不得只根据文件名猜测。

按需读取：

```bash
git diff --cached --stat
git diff --cached
git diff --stat
git diff
git ls-files --others --exclude-standard
```

对大 diff 先看 `--stat` 和文件列表，再分模块读取关键 diff。优先理解行为变化、契约变化、测试变化、配置变化和生成物变化。

必须区分：

- staged 改动：已经进入 index，但不一定属于本次提交。
- unstaged 改动：工作区中尚未 stage 的修改。
- untracked 文件：新文件，必须判断是否是本次任务产物、临时文件、生成文件或其他会话文件。
- deleted 文件：删除风险更高，必须确认是否有意删除。

## 改动分类

把所有改动分成三类：

- 本次任务相关改动：明显服务于当前用户请求或当前上下文。
- 可能相关改动：看起来有关，但无法确认是否应一起提交。
- 无关或其他会话改动：和当前目标不一致、位于明显不同模块、来自其他 skill/会话、或像用户并行编辑。

如果存在“可能相关”或“无关或其他会话改动”，提交前必须暂停并询问用户。

询问时列出：

- 建议纳入本次提交的文件。
- 不建议纳入或不确定的文件。
- 每组文件的判断原因。
- 可选决策：只提交相关改动、一起提交全部改动、按用户指定文件提交、取消提交。

不得在未确认时把无关改动一起提交。

## Staged 区域规则

已 staged 不等于用户授权提交。

如果 staged 区域包含不确定或无关改动：

- 不得直接提交。
- 先说明 staged 中有哪些不确定内容。
- 询问用户是否保留在本次提交中。
- 若用户选择排除，执行任何 `git restore --staged`、重新 stage 或调整 index 的命令前，必须说明会改变 index 状态并获得确认。

只 stage 用户确认纳入本次提交的文件。不要使用 `git add .`，除非用户明确要求提交全部改动且已确认无关改动风险。

## 提交信息生成

生成提交信息时综合：

- 用户请求和当前任务上下文。
- 实际 diff 中体现的行为变化。
- 最近 20 条提交标题。
- 项目提交规范和任务编号规则。
- 已运行验证和未验证内容。

优先遵守项目规范。没有明确规范时，默认使用简洁中文提交标题。

若历史明显使用 Conventional Commits，使用：

```text
type(scope): summary
```

若历史明显使用中文自然语言，使用一句清晰中文标题。若项目历史要求工单号或 TAPD/Jira/Linear 编号，按历史格式补充。

提交信息要求：

- 首行说明做了什么，避免“更新代码”“修复问题”“调整文件”。
- 不夸大 diff 中没有体现的行为。
- 多模块改动使用正文说明要点。
- 重大行为、契约、配置、测试变化要在正文中写清。
- 如果项目历史不记录验证，不要强行把验证写进 commit message；验证结果放最终回复即可。

## 提交流程

只有用户明确要求提交时执行：

1. 重新运行 `git status --short --branch`，确认没有新的意外改动。
2. 对照已确认的提交范围，只 stage 对应文件。
3. 执行 `git diff --cached --stat`，必要时读取 `git diff --cached` 复核将提交内容。
4. 生成最终提交信息。
5. 如果用户要求“直接提交”或已经确认提交信息，执行 `git commit`。
6. 提交后运行 `git status --short --branch`，确认提交结果和剩余改动。

多行提交信息使用多个 `-m`：

```bash
git commit -m "提交标题" -m "提交正文"
```

如果提交失败，读取错误信息并说明原因。不得通过跳过 hook、删除 hook、隐藏错误或绕过校验来让提交成功，除非用户明确要求并确认风险。

## 推送流程

只有用户明确要求推送时执行：

1. 提交成功后确认当前分支和 upstream：`git status --short --branch`。
2. 如果已有 upstream，执行 `git push`。
3. 如果没有 upstream，读取 `git remote -v`，说明当前分支、可用 remote 和建议命令，询问是否使用 `git push -u <remote> <branch>`。
4. 推送失败时，读取错误信息并说明原因、影响和下一步。

禁止执行 `git push --force` 或 `git push --force-with-lease`，除非用户明确要求强推，并在执行前再次确认风险、目标分支和回滚方式。

## 验证要求

提交前优先复用上下文中已经完成的测试、类型检查、lint、构建、手工验证或 review 结论。

如果没有验证证据：

- 不要声称已验证。
- 可以根据项目文档和改动范围建议最小验证命令。
- 未经用户要求，不运行未知脚本、安装依赖或执行可能改变外部状态的命令。

提交后至少用 `git status --short --branch` 验证 Git 状态。推送后说明远端分支和命令结果。

## 停止条件

遇到以下情况必须暂停并询问用户：

- 工作区包含明显无关或其他会话改动，且用户没有明确提交范围。
- staged 区域包含不确定或无关内容。
- 需要调整 index 以排除已 staged 内容。
- 提交规范要求的信息缺失，例如工单号、变更类型、scope。
- 没有 upstream，需要设置上游分支。
- 需要 force push、跳过 hook、删除文件、恢复文件、清理 untracked 文件或执行其他破坏性操作。
- 当前分支、远端或目标环境不明确。

## 最终回复

完成后简洁汇报：

- 提交范围：纳入了哪些文件或模块。
- 提交哈希：如已提交。
- 提交信息：实际使用的标题和正文。
- 推送结果：是否已推送，推送到哪个远端和分支。
- 剩余改动：未提交的 staged、unstaged、untracked 或明确排除的文件。
- 验证证据：运行过什么，结果如何。
- 未验证内容：无法验证或未执行的检查，以及剩余风险。

如果只生成提交信息，输出推荐提交信息、依据、建议纳入的文件和不应纳入的文件。

只生成提交信息且工作区混有无关或不确定改动时，提交信息必须基于“建议纳入范围”，同时列出“不应纳入/不确定范围”。不得生成一个看起来覆盖全量工作区的提交信息，除非用户明确要求全量提交且已确认无关改动风险。
