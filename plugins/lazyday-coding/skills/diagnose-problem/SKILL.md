# 问题定位

## 目标

定位问题根因，而不是快速猜一个看似合理的原因。

必须结合用户提供的资料和当前项目上下文建立证据链，尽可能还原问题发生过程、关键时间线、问题标识、影响范围、触发条件和代码路径。所有结论都必须有依据；证据不足时明确标注，不得把猜测包装成事实。

本 skill 是独立工具，可以单独用于 bug 定位、线上事故分析、测试失败分析、日志分析、回归问题分析、偶发问题排查和根因报告输出。它只负责定位和分析，可以给出修复建议，但不负责直接改代码。用户明确要求修复时，再交给 `implement-change` 或按当前环境规则进入实现；需要从已有产物恢复问题上下文时，先交给 `resume-context`。

## 默认模式

默认只读。

允许：

- 阅读用户资料、日志、错误、截图描述、测试输出和代码。
- 使用只读命令搜索、查看文件、查看 Git 历史、查看 diff、梳理调用链。
- 在安全前提下运行已有的复现命令、测试、类型检查、lint、构建或 smoke，用于验证假设。
- 输出阶段性分析、问题报告、根因排名和修复建议。

禁止：

- 未经明确要求修改、创建、删除文件。
- 安装依赖、升级依赖、切换分支、清理工作区、提交代码或执行破坏性命令。
- 在证据不足时给确定结论。
- 只通过吞异常、放宽断言、跳过测试、增加无依据 retry/timeout 或隐藏错误来解释问题。

如果复现、测试或诊断命令可能改变外部状态、访问生产资源、耗时很长或产生大量文件，先说明原因并请求确认。

## 产物交接

需求产物由 `manage-artifacts` 统一管理。需要创建、选择、切换、列出、归档或修复 `.lazyday/coding/requirements/` 时，先使用 `manage-artifacts`。如果用户说“继续”“上次”“当前需求”但没有明确需求 ID，先使用 `resume-context` 匹配候选需求。

当前需求已明确时，本 skill 只读取和更新与问题定位相关的产物。

本 skill 优先读取：

- `brief.md`：用户目标、期望行为、实际行为和成功标准。
- `repo-context.md`：相关入口、调用链、日志关键词、测试位置和风险边界。
- `verification.md`：失败命令、失败输出和未通过验证证据。
- `review.md`：review 中发现的疑似缺陷或回归风险。
- `handoff.md`：上一阶段最短交接摘要。

当用户要求跨会话记录、继续交接或允许落盘，且当前需求已明确时，把问题定位报告写入 `diagnosis.md`，并同步更新 `handoff.md`。不要默认提交 `.lazyday/`。

## 输入处理

优先提取并保留用户提供的原始资料：

- bug 描述、期望行为、实际行为。
- 报错文本、堆栈、日志、截图、录屏、告警。
- 发生时间、最近正常时间、版本、分支、commit、环境、账号、设备、地域、配置、数据样本。
- 复现步骤、触发频率、是否必现、是否回归、最近变更。
- request id、trace id、session id、job id、user id、message id、error code、feature flag、版本号等问题标识。
- 用户认为相关的文件、PR、提交、发布、配置或任务。

如果资料不足，先基于当前项目做只读探索；仍不足以支撑判断时，向用户索要最小必要资料。不要一次性索要过多信息，优先问能最大幅缩小定位范围的问题。

## 执行流程

### 1. 问题复述

先用自己的话复述：

- 用户报告的问题是什么。
- 期望行为和实际行为分别是什么。
- 已知事实、未知信息和当前矛盾点是什么。
- 当前阶段是只读定位、复现验证，还是用户已授权修复。
- 哪些结论暂时不能下。

### 2. 时间线重建

从用户资料和项目证据中提取时间线：

- 问题第一次出现时间。
- 最近确认正常时间。
- 用户操作顺序、系统事件顺序、异步任务顺序。
- 相关发布、提交、配置、数据、依赖或环境变更。
- 日志中的关键标识和状态转换。

没有明确时间线时，分成“已知时间线”和“缺失时间线”。推断出的节点必须标注“推断”或“待验证”。

### 3. 上下文收集

在仓库中展开只读探索：

- 查找并遵守项目级、目录级 `AGENTS.md` 和用户当前指令。
- 检查相关文件、调用点、类型、配置、测试、fixture、mock 和文档。
- 搜索错误码、日志关键词、函数名、接口名、事件名、字段名和问题标识。
- 梳理入口、调用链、数据流、状态归属、错误处理、重试、缓存、并发、资源清理和降级路径。
- 检查相关历史变更：`git status`、`git diff`、`git log`、`git blame`、测试快照、配置变化和依赖变化。
- 对前后端、后端、Electron、文件系统、数据库、并发、安全、性能、发布等问题域，只展开与当前问题相关的专项检查。

不得只看一个局部文件就下结论。当前证据不能解释现象时，必须扩大分析范围。

### 4. 候选根因生成

列出多个候选根因。每个候选都必须包含：

- 假设内容。
- 支持证据。
- 反向证据。
- 需要验证的代码路径、数据路径或运行路径。
- 可执行的验证动作。
- 当前置信度。

候选根因至少按相关性考虑：

- 代码逻辑错误。
- 调用契约、类型、序列化、数据格式不一致。
- 边界条件、空值、异常状态或旧数据。
- 配置、环境、版本、feature flag 或发布差异。
- 异步、并发、时序、缓存、重入、取消、stale response。
- 错误处理、降级、重试、超时、资源清理。
- 依赖、构建、平台、打包或运行时差异。
- 测试、mock、fixture 与真实行为不一致。
- 用户描述、日志来源或环境标识本身存在偏差。

### 5. 多轮验证

对候选根因逐个验证或反证：

- 静态证据：代码、类型、配置、调用链、历史 diff。
- 动态证据：复现命令、测试、脚本、日志过滤、最小输入样例。
- 对照证据：正常路径 vs 异常路径，旧版本 vs 新版本，mock vs 真实实现。
- 反事实检查：如果假设成立，应该还能观察到什么；如果观察不到，降低置信度。

一轮验证不足时继续扩大范围。不要因为单一路径走不通就停住；要切换到其他候选和其他证据来源。

### 6. 矛盾处理

发现用户描述、日志、代码和推理之间存在矛盾时：

1. 先检查是否读错日志、路径、版本、分支、环境、时间顺序或问题标识。
2. 再检查是否存在多问题叠加、旁路失败、旧日志和新日志混杂、不同环境混杂。
3. 仍冲突时，明确指出“当前证据与用户描述不一致”，并说明可能需要修正的时间、环境、版本、触发条件或现象边界。

质疑用户描述前，必须先质疑自己的分析链。

### 7. 停止条件

可以停止并输出报告的条件：

- 已找到强证据支撑的根因。
- 已排除主要候选，仅剩需要外部资料验证的候选。
- 当前环境无法继续验证，且已说明缺失资料和下一步最小取证动作。
- 用户要求先输出阶段性分析。

不能停止的情况：

- 只有一个未经验证的猜测。
- 证据只能解释部分现象，但没有说明剩余矛盾。
- 只检查了表层报错，没有检查触发路径和上游原因。
- 一条路径走不通后没有尝试替代路径。

## 证据和置信度

需要更细的评分边界、证据强度和候选根因排序时，读取 `references/diagnosis-rubric.md`。

根因置信度使用 0-100 分：

- 90-100：已复现，或日志、测试、代码路径直接证明。
- 70-89：多条强相关证据支持，反向证据较少，但缺少完整复现。
- 40-69：有合理证据，但关键验证缺失或仍有明显竞争假设。
- 1-39：仅为排查方向或弱假设。
- 0：已被证据排除。

输出时区分：

- 事实：来自用户资料、代码、日志、测试、配置、版本或可靠来源。
- 推断：基于事实的合理解释。
- 假设：尚未验证的候选方向。
- 建议：基于当前证据的下一步动作。

## 质量门

输出前自检：

- 是否遵守只读边界。
- 是否完整提取了用户资料中的时间、版本、环境、错误和问题标识。
- 是否还原了时间线，而不是只解释最终报错。
- 是否读取了足够的项目上下文和相关代码实现。
- 是否列出多个候选根因，并尝试验证或排除。
- 是否把事实、推断、假设和建议区分清楚。
- 是否说明了与用户描述矛盾的地方。
- 是否对每个重要结论给出证据来源。
- 是否给出根因置信度评分和排名。
- 如果没有找到根因，是否说明已尝试路径、排除依据、剩余候选和下一步取证动作。

## 后续交接

- 需要从已有产物恢复上下文或匹配候选需求：交给 `resume-context`。

- 需要方案取舍：交给 `research-approach`，附带当前根因候选和约束边界。
