# 问题定位评分参考

本参考用于 `diagnose-problem` 需要细化证据强度、根因置信度和候选根因排名时读取。

## 证据强度

### 强证据

强证据可以直接证明、复现或反证候选根因：

- 稳定复现步骤和对应失败输出。
- 测试、脚本、日志过滤或最小输入样例证明某条路径必然失败。
- 代码路径直接展示错误状态如何产生。
- 同一 request id、trace id、job id、session id 下的完整事件链。
- 明确的版本对照：旧版本正常，新版本异常，且差异与候选根因相关。
- 与真实运行环境一致的配置、数据、依赖和平台证据。

### 中证据

中证据高度相关，但不能单独证明根因：

- 可疑代码、配置或历史变更与现象方向一致。
- 日志中存在同类错误，但缺少完整上下游链路。
- 测试或 mock 能触发相似问题，但与真实场景不完全一致。
- 调用链或数据流显示存在缺口，但缺少实际触发证据。
- 用户描述与某个候选根因高度吻合，但缺少环境或版本证据。

### 弱证据

弱证据只能作为排查线索：

- 经验判断。
- 命名、模块位置或相似历史问题。
- 不完整日志片段。
- 无法确认环境、版本、输入和时间的现象描述。
- 模型推断。

弱证据不能支撑确定结论，只能用于生成候选假设。

## 根因置信度

### 90-100：确定或接近确定

满足以下条件中的多数：

- 已在当前环境或等价环境复现。
- 代码路径、日志、输入和输出形成闭环。
- 已排除主要竞争假设。
- 修复方向可以由证据直接推出。
- 没有明显未解释矛盾。

### 70-89：高可信

满足以下条件中的多数：

- 多条强证据或强中混合证据指向同一原因。
- 有清晰时间线和问题标识。
- 主要代码路径已验证。
- 仍缺少完整复现或真实环境确认。
- 反向证据较少，且能解释。

### 40-69：中等可信

典型状态：

- 有合理证据支持，但关键证据缺失。
- 存在多个竞争假设。
- 能解释主要现象，但解释不了全部细节。
- 需要更多日志、数据、环境或复现步骤。

### 1-39：低可信

典型状态：

- 只有弱证据或经验判断。
- 缺少代码路径验证。
- 与部分事实冲突。
- 更适合作为下一步排查方向，而不是结论。

### 0：已排除

满足任一条件：

- 代码路径无法到达。
- 版本、环境或时间线不匹配。
- 复现或测试反证。
- 日志或数据直接证明该路径未发生。

## 排名方法

候选根因排名时综合考虑：

- 证据强度。
- 能解释的现象比例。
- 与时间线的匹配程度。
- 与代码路径和数据路径的匹配程度。
- 是否能解释用户看到的最终表现。
- 是否存在未解释矛盾。
- 是否已有复现或反证。
- 修复建议是否能直接从该根因推出。

不要只按“最常见原因”排名。常见但无证据的原因必须低于少见但证据更强的原因。

## 常见误判

- 把最终报错当成根因，忽略上游状态污染。
- 只相信用户描述，不核对环境、版本、时间和输入。
- 只看最新日志，忽略同一 trace 下更早的异常。
- 只看业务代码，忽略配置、构建、发布、缓存、依赖和数据迁移。
- 只看成功路径，忽略取消、重入、超时、降级和部分失败。
- 因为一个候选无法验证就停止，而不是切换思路。
- 用“可能是”替代证据链。

## 取证问题模板

需要向用户补充资料时，优先问能缩小范围的问题：

- 问题第一次出现的准确时间和最近正常时间是什么？
- 发生环境、版本、分支或 commit 是什么？
- 是否有 request id、trace id、用户 id、job id、错误码或完整日志？
- 是否稳定复现？最短复现步骤是什么？
- 最近是否有发布、配置、依赖、数据或权限变化？
- 同一操作是否只影响特定账号、数据、设备、地域或入口？
- 是否有正常样本和异常样本可以对照？
