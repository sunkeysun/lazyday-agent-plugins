# 项目探索

## 目标

快速建立足够准确的仓库上下文，让后续方案、拆解、实现、验证、review 和提交不再依赖猜测。重点是找到项目规则、相关入口、调用链、已有模式、测试位置、风险边界和可复用实现。

本 skill 是独立工具，可以单独用于新仓库 onboarding、陌生模块探索、实现前上下文构建、review 前范围识别和 bug 定位前调用链梳理。它不依赖其他 skill；如有需要，只在最后建议交给 `resume-context`、`research-approach`、`break-down-task`、`diagnose-problem`、`implement-change`、`verify-change`、`review-code` 或 `commit-changes`。

## 默认模式

默认只读。

允许：

- 使用 `rg`、`rg --files`、`find`、`git status`、`git diff`、`git log`、`sed` 等只读命令。
- 阅读项目规则、README、配置、测试、入口、调用方、类型、mock、fixture 和历史提交。
- 输出仓库地图、相关文件清单、调用链、验证命令和风险边界。

禁止：

- 未经明确要求修改、创建、删除代码文件。
- 安装依赖、切换分支、清理工作区、提交代码或执行破坏性命令。
- 无边界扫描整个仓库，把无关上下文塞进输出。
- 只看局部文件就宣称理解了整体路径。

当用户要求“记录探索结果”“后续继续用”“跨会话交接”但尚未选择需求产物时，先交给 `manage-artifacts` 创建或选择需求。当前需求已明确且用户允许跨会话记录或落盘时，才可以写入或更新 `repo-context.md` 和 `handoff.md`。

## 产物交接

需求产物由 `manage-artifacts` 统一管理。需要创建、选择、切换、列出、归档或修复 `.lazyday/coding/requirements/` 时，先使用 `manage-artifacts`。如果用户说“继续”“上次”“当前需求”但没有明确需求 ID，先使用 `resume-context` 匹配候选需求。

当前需求已明确时，本 skill 只读取和更新与项目探索相关的产物：

- 读取：`brief.md`、`repo-context.md`、`research.md`、`plan.md`、`diagnosis.md`、`handoff.md`。
- 写入：`repo-context.md`、`handoff.md`。

不要擅自混合多个需求的上下文。`.lazyday/` 默认是本地协作产物，不纳入业务提交。

## 探索流程

### 1. 确定探索目标

先明确探索服务于什么：

- 为方案调研找项目约束和已有模式。
- 为任务拆解找模块边界、调用方和测试位置。
- 为问题定位找入口、日志、状态流和错误路径。
- 为实现找可修改文件、可复用工具和 dirty worktree 风险。
- 为 review 找 diff 范围、核心风险和上下文文件。
- 为验证找命令、测试层级和 smoke 路径。

不要泛泛探索。每次探索必须有目标、范围和停止条件。

### 2. 项目规则与状态

先建立安全边界：

```bash
git status --short --branch
rg --files -g 'AGENTS.md' -g 'CLAUDE.md' -g 'README*' -g 'CONTRIBUTING*'
```

从相关目标路径向上读取适用的 `AGENTS.md` 或项目规则。若没有明确目标路径，先读取仓库根规则和 README。

记录：

- 当前分支和 dirty worktree。
- 适用规则文件。
- 构建、测试、lint、类型检查入口。
- 禁止修改或需要确认的区域。

### 3. 结构地图

按目标选择最小范围：

- 先用 `rg --files` 和目录清单识别模块。
- 再搜索关键词、类型、函数、接口、错误码、路由、事件、配置和测试名。
- 对候选文件先看文件名、导出、调用点和测试，再决定是否深入。
- 对大仓库先输出探索地图，不要读全量文件。

### 4. 调用链与数据流

根据问题域梳理：

- 入口：用户操作、API、CLI、任务、事件、IPC、hook、路由。
- 核心链路：主要函数、服务、状态更新、数据读写、外部依赖。
- 旁路链路：日志、缓存、遥测、异步任务、降级、重试。
- 错误路径：异常捕获、错误码、提示、回滚、资源清理。
- 测试路径：单测、集成、E2E、fixture、mock 和手工 smoke。

### 5. 复用点和风险点

输出时必须列出：

- 已有可复用实现。
- 已有约定和命名模式。
- 相关测试与验证命令。
- 可能冲突的用户改动。
- 核心链路和高风险文件。
- 仍未探索或不确定的路径。

## 后续交接

- 需要从已有产物恢复上下文或匹配候选需求：交给 `resume-context`。
- 需要创建、选择或切换需求产物：交给 `manage-artifacts`。
- 需要方案：交给 `research-approach`，附带 `brief.md` 和 `repo-context.md`。
- 需要拆任务：交给 `break-down-task`，附带相关文件、调用链、测试位置和风险边界。
- 需要定位问题：交给 `diagnose-problem`，附带入口、日志关键词、调用链和候选文件。
- 需要实现：交给 `implement-change`，附带允许修改范围、可复用实现和验证命令。
- 需要验证：交给 `verify-change`，附带验证入口和预期证据。
- 需要 review：交给 `review-code`，附带高风险文件和上下文地图。
