# 代码实现

## 目标

在用户已明确授权修改代码时，把需求、方案、任务卡或修复建议落到真实仓库改动中。核心目标是最小、正确、可验证，而不是扩大范围或顺手重构。

## 进入条件

只有用户明确要求实现、修改、修复、落地、按方案执行、改代码或处理问题时才使用本 skill。用户只是要求分析、review、出方案、评估、解释原因或“先看看”时，保持只读并转向 `research-approach`、`diagnose-problem` 或 `review-code`。如果用户说“继续实现”“接着做”但需求上下文不明确，先使用 `resume-context`。

## 基线协议

实现前必须完成：

1. 读取用户最新指令，确认当前不是只读任务。
2. 从目标文件目录向上查找项目级或目录级 `AGENTS.md`。
3. 运行 `git status --short --branch` 建立 dirty worktree 基线。
4. 对将要修改的文件检查当前内容和相关 diff。
5. 搜索已有实现、调用点、类型、测试和项目约定。

发现相关文件已有未提交改动时，先读懂 diff。无法判断如何安全合并时，暂停并让用户确认。

## 跳步执行

用户可以跳过澄清、探索、方案和拆解，直接要求实现。此时不要强制补跑所有前序 skill，但必须在实现前内部补齐最小执行契约：

- 本轮目标：要实现或修复什么。
- 非目标：明确不做什么，尤其是不顺手重构、不扩大范围。
- 授权状态：用户是否明确允许修改文件。
- 修改边界：预计会读写哪些模块或文件，哪些不能碰。
- 风险等级：是否触发公共契约、数据、权限、依赖、发布或核心链路确认门。
- 复用点：项目已有实现、工具、类型、测试和错误处理模式。
- 验证计划：最小可行验证命令或手工证据。

如果这些信息可通过用户请求、当前代码和只读探索补齐，直接继续实现。如果关键信息不足且会影响核心链路、高风险决策或确认门，先问最小确认问题。

跳步实现不要求创建 `.lazyday/coding/requirements/`。只有用户要求跨会话记录、完整需求管理或后续交接时，才写入实现产物。

## 产物交接

需求产物由 `manage-artifacts` 统一管理。需要创建、选择、切换、列出、归档或修复 `.lazyday/coding/requirements/` 时，先使用 `manage-artifacts`。如果用户说“继续”“上次”“当前需求”但没有明确需求 ID，先使用 `resume-context` 匹配候选需求。

当前需求已明确时，本 skill 只读取和更新与代码实现相关的产物。

本 skill 优先读取：

- `brief.md`：目标、非目标、验收标准、风险等级和确认门。
- `repo-context.md`：项目规则、允许修改范围、相关入口、可复用实现和验证命令。
- `research.md`：推荐方案、范围边界和不做事项。
- `plan.md`：任务卡、文件所有权、执行步骤和验证方式。
- `diagnosis.md`：bug 根因、最小修复建议和原失败路径。
- `handoff.md`：上一阶段最短交接摘要。

当用户要求跨会话记录、继续交接或允许落盘，且当前需求已明确时，把实现摘要写入 `implementation.md`，并同步更新 `handoff.md`。不要默认提交 `.lazyday/`。

## 任务图执行

当存在 `plan.md`、任务卡、执行调度计划或用户指定 task 时，必须按任务图执行，而不是按普通实现请求自由发挥。

### 1. 选择当前任务

先读取并整理：

- 任务依赖图。
- 覆盖矩阵。
- 冲突矩阵。
- 执行调度计划。
- 并发批次。
- Agent Packets。
- 任务卡。
- 验证矩阵。

选择当前任务时按优先级：

1. 用户明确指定的 task。
2. `ready_now` 中风险最低、阻塞最多下游的 task。
3. 当前批次中与已有 dirty worktree 冲突最少的 task。
4. 如果所有 task 都被阻塞，先输出阻塞原因和最小解阻动作，不要擅自跳到下游 task。

不得执行未满足 `blocks`、`confirm-gate`、`review-gate` 或 `merge-gate` 的任务。不得把多个任务合并实现，除非它们文件所有权一致、依赖已满足、验证可以一起覆盖，并且不会扩大范围。

### 2. 依赖与冲突门

写代码前必须检查：

- 当前 task 的依赖是否完成。
- 当前 task 是否触发用户确认门。
- 当前 task 的 `allowed_write_paths` 和 `forbidden_write_paths`。
- 当前 dirty worktree 是否已经修改了这些路径。
- 冲突矩阵中的共享资源是否已有 owner。
- 是否会碰 lockfile、生成物、schema、barrel export、配置入口、路由入口、公共类型或发布脚本。

如果目标文件已有用户或其他 agent 改动，先读取 diff 并判断是否可安全叠加。不能安全叠加时暂停请求确认，不要覆盖。

### 3. 并发执行策略

按最高效且安全的方式选择执行模式：

- **single-agent sequential**：存在共享文件、公共契约、dirty worktree 冲突、验证互相污染或高风险核心链路时使用。
- **single-agent staged**：任务可分阶段完成，但每阶段后需要 merge gate 或验证 gate 时使用。
- **multi-agent patches**：任务文件所有权不重叠，但共享工作区风险较高时使用；其他 agent 只交付补丁和验证摘要，由主 agent 合并。
- **multi-worktree**：高价值并发且任务写入范围独立、验证成本高或共享 dirty worktree 风险不可接受时建议；未获用户明确授权前不要创建 worktree。

只有满足以下条件才允许并发写入：

- 每个 agent 的写入范围完全不重叠。
- 不共享 lockfile、生成物、schema、barrel export、公共配置或入口注册文件。
- 不依赖同一个未稳定契约。
- 验证命令不会互相污染缓存、数据库、端口或生成物。
- 有明确 merge owner 和每次合并后的验证命令。

不满足这些条件时，可以并发只读探索、并发生成补丁建议，不能并发写同一工作区。

### 4. 合并门

完成一个 task 或合并一个 agent 输出后，必须：

1. 重新运行 `git status --short --branch`。
2. 检查本 task 涉及文件的 diff。
3. 确认没有触碰禁止范围。
4. 运行该 task 的局部验证或说明无法运行原因。
5. 更新当前 task 状态、剩余 blocked task、下一批 ready task。

进入下一批任务前，按 `merge_sequence` 执行 merge gate。merge gate 未通过时，不要继续下游实现。

## 实现规则

- 每个改动都必须服务当前请求。
- 优先复用已有函数、类型、组件、服务、测试工具和错误处理模式。
- 不新增依赖、不改公共契约、不删代码、不做迁移、不动发布配置，除非用户明确确认。
- 涉及 bug 修复时，先保留失败证据和根因假设，再改代码。
- 涉及异步、文件系统、网络、进程、权限或核心链路时，检查竞态、幂等、取消、超时、资源清理和失败隔离。
- 写入前重新读取目标片段；写入后检查 diff，确认没有无关改动。
- 按任务卡执行时，只修改 `allowed_write_paths`；确需超出范围时先说明原因并确认。

## 验证

优先运行与改动最贴近的验证：

1. 目标测试或最小复现。
2. 类型检查、lint 或构建。
3. 集成、E2E 或 smoke。
4. 静态审查和手工路径说明。

不能运行验证时，必须说明原因、替代证据和剩余风险。不要用“应该可以”代替证据。

## 输出

简洁汇报：

- 完成了什么。
- 执行了哪些 task，哪些 task 仍 blocked。
- 是否按调度计划执行；如果偏离，说明原因。
- 修改了哪些文件。
- 是否触碰共享冲突点或禁止范围。
- 运行了什么验证和结果。
- 未验证内容和剩余风险。
- 下一批 ready task 或 merge gate 状态。
- 若本轮是单点实现：明确本轮到实现和验证结果即完成，不要求继续其他 skill。
- 若本轮属于完整需求：说明已更新或建议更新的阶段产物、`handoff.md` 和下一步 skill。
