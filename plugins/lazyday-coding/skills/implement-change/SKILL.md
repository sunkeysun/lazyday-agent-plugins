# 代码实现

## 目标

在用户已明确授权修改代码时，把需求、方案、任务卡或修复建议落到真实仓库改动中。核心目标是最小、正确、可验证，而不是扩大范围或顺手重构。

## 进入条件

只有用户明确要求实现、修改、修复、落地、按方案执行、改代码或处理问题时才使用本 skill。用户只是要求分析、review、出方案、评估、解释原因或“先看看”时，保持只读并转向 `research-approach`、`diagnose-problem` 或 `review-code`。

## 基线协议

实现前必须完成：

1. 读取用户最新指令，确认当前不是只读任务。
2. 从目标文件目录向上查找项目级或目录级 `AGENTS.md`。
3. 运行 `git status --short --branch` 建立 dirty worktree 基线。
4. 对将要修改的文件检查当前内容和相关 diff。
5. 搜索已有实现、调用点、类型、测试和项目约定。

发现相关文件已有未提交改动时，先读懂 diff。无法判断如何安全合并时，暂停并让用户确认。

## 实现规则

- 每个改动都必须服务当前请求。
- 优先复用已有函数、类型、组件、服务、测试工具和错误处理模式。
- 不新增依赖、不改公共契约、不删代码、不做迁移、不动发布配置，除非用户明确确认。
- 涉及 bug 修复时，先保留失败证据和根因假设，再改代码。
- 涉及异步、文件系统、网络、进程、权限或核心链路时，检查竞态、幂等、取消、超时、资源清理和失败隔离。
- 写入前重新读取目标片段；写入后检查 diff，确认没有无关改动。

## 验证

优先运行与改动最贴近的验证：

1. 目标测试或最小复现。
2. 类型检查、lint 或构建。
3. 集成、E2E 或 smoke。
4. 静态审查和手工路径说明。

不能运行验证时，必须说明原因、替代证据和剩余风险。不要用“应该可以”代替证据。

## 输出

简洁汇报：

- 完成了什么。
- 修改了哪些文件。
- 运行了什么验证和结果。
- 未验证内容和剩余风险。
