# 方案调研

## 目标

产出证据充分、复杂度合适、可实施、可交给后续实现 agent 的技术方案。重点不是“给一个看似正确的方案”，而是先确认用户真正要解决的问题、当前项目阶段、生产约束和可接受复杂度，再给出几种同时满足需求但侧重点不同的方案供用户决策。

本 skill 是独立工具，可以单独用于方案调研、技术选型、架构评估、实现路径设计和方案 review。它不依赖其他 skill；如有需要，只在最后建议后续交给 `resume-context`、`break-down-task`、`implement-change`、`verify-change` 或 `review-code`。

## 工作模式

默认只读。

禁止修改文件、创建文件、安装依赖、执行破坏性命令、提交代码或改变外部状态。可以使用只读命令、项目上下文探索、记忆检索、官方文档、源码、release note、标准规范、网络搜索和用户澄清来补足证据。

当用户明确要求“把方案写入文件/创建文档/落地到 skill”时，才允许进入写入模式；写入前仍要检查项目规则和 dirty worktree。

## 产物交接

需求产物由 `manage-artifacts` 统一管理。需要创建、选择、切换、列出、归档或修复 `.lazyday/coding/requirements/` 时，先使用 `manage-artifacts`。如果用户说“继续”“上次”“当前需求”但没有明确需求 ID，先使用 `resume-context` 匹配候选需求。

当前需求已明确时，本 skill 只读取和更新与方案调研相关的产物。

本 skill 优先读取：

- `brief.md`：任务目标、非目标、验收标准、风险等级和确认门。
- `repo-context.md`：项目规则、入口、调用链、测试位置和可复用模式。
- `diagnosis.md`：当方案来自 bug 根因或修复建议时读取。
- `handoff.md`：读取上一阶段给出的最短交接摘要。

当用户要求跨会话记录、继续交接或允许落盘，且当前需求已明确时，把方案调研结果写入 `research.md`，并同步更新 `handoff.md`。不要默认提交 `.lazyday/`。

## 理解门槛

设计方案前，必须先建立足够上下文。至少确认：

- 用户真正要解决的问题，而不是只复述表面需求。
- 明确目标、非目标和验收标准。
- 当前阶段：探索、MVP、生产修复、架构演进、重构、发布前加固。
- 目标用户、运行环境、调用方和交付形态。
- 当前项目已有实现、约定、限制和可复用能力。
- 核心链路、旁路链路、失败后果和回滚要求。
- 风险等级、相关问题域和生产常见坑。
- 用户可接受的复杂度、工作量、上线节奏和长期演进预期。

如果关键信息不足，必须先补上下文，不能直接给最终方案。补上下文按成本从低到高选择：

1. 向用户提出 1-3 个高价值澄清问题。
2. 读取项目级/目录级规则、README、设计文档和相关代码。
3. 搜索已有实现模式、调用方、测试、配置和历史兼容路径。
4. 搜索记忆，查找用户偏好、项目先例和之前决策。
5. 查官方文档、源码、release note、标准规范和维护者说明。
6. 对最新能力、依赖行为、框架规则或行业实践进行联网确认。
7. 若仍无法补齐，明确列出假设、缺口和这些缺口如何影响方案可靠性。

低风险任务可以带着显式假设继续；中高风险或核心链路任务在关键问题未明确前，不要给“最终推荐方案”。

如果缺少真实仓库上下文、`repo-context.md`、实际代码调用链或测试入口，方案仍可作为初步建议输出，但必须降低推荐置信度，列出关键假设，并说明哪些结论需要通过项目探索或代码读取确认后才能交给实现。

## 问题域识别

先判断任务属于哪些问题域，并只展开相关检查：

- 前端/UI：状态归属、loading/error/empty、交互流畅性、可访问性、响应式、视觉回归。
- 后端/API：契约兼容、权限、幂等、事务、分页、限流、错误码、超时、重试。
- Electron/桌面端：main/preload/renderer 边界、IPC、跨平台路径、打包路径、自动更新、退出清理。
- 文件系统/下载/安装：路径穿越、原子写入、staging、hash/signature、失败清理、回滚。
- 数据库/schema/迁移：旧版本兼容、expand/contract、备份、幂等、锁风险、索引和回滚。
- 异步/并发/任务调度：竞态、重入、取消、幂等、超时、退避、stale response、资源释放。
- 安全/隐私/权限：输入校验、权限边界、注入、敏感日志、最小权限、审计。
- 性能/资源：复杂度、内存、CPU、I/O、缓存、懒加载、批量处理和回退方案。
- Dev tooling/构建/CI：命令契约、缓存、跨平台、lockfile、生成物、CI 可重复性。
- AI agent/工作流/harness：阶段门、上下文压缩、工具权限、diff 边界、多 agent 协作、验证闭环。
- 插件/扩展架构：manifest、加载生命周期、版本兼容、权限声明、宿主边界、分发更新。
- 发布/灰度/兼容性：feature flag、kill switch、旧版本兼容、监控、回滚和通知。

需要更完整的领域风险、评分细则或 AI 工作量估算时，读取 `references/assessment-rubric.md`。

## 证据规则

优先使用高质量证据，不要把模型推断包装成事实。

证据优先级：

1. 当前项目代码、项目规则、测试、配置和真实调用链。
2. 官方文档、标准规范、源码、release note、迁移指南。
3. 维护者 issue、RFC、设计说明和真实变更记录。
4. 生产实践、成熟工程模式和可验证案例。
5. 文章、博客、评测、社区讨论。
6. 模型经验和推断。

输出时必须区分：

- 事实：来自代码、文档、日志、测试或可靠来源。
- 推断：基于现有证据的合理判断。
- 建议：结合用户目标、项目阶段和取舍后的行动主张。

涉及最新信息、依赖行为、框架规则、模型能力、安全规则或高风险决策时，优先联网或读取官方来源确认。无法确认时明确标注。

## 方案设计规则

输出 2-4 个可行方案。每个方案都必须能满足用户核心目标，但强调不同取舍。不要凑弱方案；如果只有一个可信方案，说明为什么其他方向不成立。

常见方案类型：

- 最小局部方案：最快落地、影响面小，适合 MVP、止血或低风险需求。
- 平衡生产方案：兼顾稳定性、工作量、验证和可维护性，通常作为默认推荐。
- 强健可扩展方案：面向长期演进、复杂场景、多调用方或高风险核心链路。
- 阶段化方案：先低风险落地，再通过后续步骤扩展、迁移或加固。
- 复用现有方案：优先复用项目已有服务、模式、工具或第三方能力。
- 不做/延后方案：当需求收益不足、证据不足或复杂度不匹配时明确建议暂缓。

每个方案必须包含：

- 方案摘要。
- 实现形态和关键模块。
- 与当前项目的适配方式。
- 对现有契约、数据流、状态流或用户流程的影响。
- 稳定性、流畅性、安全性、可观测性、性能和回滚考虑。
- 测试与验证策略。
- AI 编程工作量估算。
- 优点、缺点、适用场景、不适用场景。

## 工作量评估

按现代全 AI 编程流程估算，不要只按代码行数判断。

必须拆分这些成本：

- 上下文发现：理解仓库、规则、调用链、现有模式和约束。
- 方案细化：确定契约、数据流、状态流、失败行为和边界。
- 代码生成：由 AI 生成主要实现的成本，通常不是最大项。
- 集成修改：接入现有模块、更新调用方、类型、mock、fixture、配置。
- 测试编写：单测、集成、E2E、回归、边界、失败路径。
- 验证执行：运行测试、构建、lint、手工 smoke、日志或截图验证。
- Review 与加固：自审、代码 review、修复反馈、安全和性能检查。
- 发布准备：灰度、回滚、迁移、文档、监控和 handoff。

使用相对规模：

- XS：局部只读结论或单点小改，分钟级。
- S：单模块内清晰改动，少量测试。
- M：多个文件或一个模块边界，存在集成验证。
- L：跨模块、状态流、核心路径或公共契约，验证成本较高。
- XL：迁移、安全、权限、发布、兼容或不可逆风险，需要分阶段和确认门。

说明工作量分布，而不是只给一个总尺寸。复杂度通常藏在上下文、集成和验证里。

## 决策矩阵

对每个方案从 1-5 分评估，必要时只解释影响决策的关键分数：

- 需求匹配度。
- 项目适配度。
- 流程稳定性。
- 用户体验或开发体验流畅性。
- 安全性和隐私。
- 实现工作量。
- 运维和可观测性复杂度。
- 未来扩展性。
- 回滚简单度。
- 验证信心。

分数不是为了显得客观，而是帮助用户看清取舍。若评分缺少证据，明确标注低置信度。

## 推荐规则

必须推荐一个最适合当前上下文的方案，除非关键上下文缺失到无法可靠推荐。

推荐时说明：

- 为什么它最符合当前阶段。
- 为什么没有过度设计。
- 为什么没有低估生产场景。
- 它刻意不做什么。
- 哪些内容应该延后。
- 实现前还需要确认什么。
- 后续实现 agent 应该优先保护哪些边界。

“最适合”优先于“最完整”。如果用户当前目标是 MVP，不要推荐重型架构；如果触达核心链路、安全、数据或发布，不要推荐只靠简单 patch 的方案。

## 质量门

输出前自检：

- 是否遵守只读边界。
- 是否真正理解用户要的方案，而不是直接回答表面问题。
- 是否探索了足够的项目上下文和已有模式。
- 是否识别了相关问题域和生产坑。
- 是否使用了足够高质量的证据，并标注事实/推断/建议。
- 是否给出 2-4 个真正可行且侧重点不同的方案。
- 是否说明每个方案适合和不适合的场景。
- 是否按 AI 编程流程拆分工作量。
- 是否明确推荐最适合当前阶段的方案。
- 是否给后续实现 agent 留下清晰、可执行、可验证的指导。

## 后续交接

- 需要从已有产物恢复上下文或匹配候选需求：交给 `resume-context`。
- 需要创建、选择或切换需求产物：交给 `manage-artifacts`。
- 需要澄清需求或选择需求产物：交给 `clarify-task`。
- 需要补项目上下文：交给 `explore-repo`。
- 需要拆任务：交给 `break-down-task`，附带 `brief.md`、`repo-context.md` 和 `research.md`。
- 需要实现：交给 `implement-change`，附带推荐方案、范围边界和验证计划。
- 需要验证方案或实现结果：交给 `verify-change`。
- 需要审查已有方案或 diff：交给 `review-code`。
