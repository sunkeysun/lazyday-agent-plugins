# 方案调研评估细则

本文件用于 `research-approach` 的深度方案调研。只有在用户要求“深入、生产级、最佳实践、完整方案、多方案对比”，或任务风险达到 S2+，或涉及核心链路、安全、数据、发布、迁移、并发、架构边界时读取。

## 上下文完整性评分

给最终方案前，先检查上下文是否足够：

- 需求清晰度：目标、非目标、验收标准是否明确。
- 项目理解：入口、调用链、已有模式、约束和测试是否清楚。
- 用户阶段：MVP、生产修复、架构演进、长期平台化是否明确。
- 风险边界：核心链路、失败后果、回滚方式和确认门是否清楚。
- 证据质量：是否有项目证据、官方证据或可验证行业实践。

若任一关键项不足，输出必须包含“缺失上下文与假设”，并说明这些缺口如何影响推荐可靠性。

## 问题域风险清单

### 前端 / UI

重点关注：

- 状态归属：数据源、缓存、派生状态、表单状态是否清楚。
- loading/error/empty：是否覆盖初始加载、刷新、失败、空数据和部分失败。
- 交互流畅性：重复点击、禁用态、乐观更新、撤销、重试、取消。
- 异步竞态：stale response、组件卸载、请求乱序、刷新覆盖。
- 权限与路由：直接访问、深链、刷新、权限变更、未授权态。
- 可访问性与响应式：键盘、焦点、语义、移动端和长文本。
- 视觉验证：是否需要截图、设计稿对照、跨视口检查。

常见坑：

- 只处理 happy path，没有失败态和重入保护。
- UI 隐藏权限入口，但后端/API 没有权限保护。
- 旧请求返回覆盖新状态。
- 组件局部实现绕过项目已有组件、表单或请求封装。

### 后端 / API

重点关注：

- 契约兼容：请求/响应结构、默认值、错误码、分页和排序。
- 权限与审计：鉴权、授权、租户隔离、操作记录。
- 幂等与事务：重复请求、部分失败、补偿、重试安全。
- 外部依赖：超时、退避、降级、熔断、错误分类。
- 数据一致性：并发写、唯一约束、乐观锁、缓存一致性。
- 观测：日志关键词、trace id、指标、告警和错误上下文。

常见坑：

- 修改返回结构但漏更新调用方、mock、fixture 或文档。
- 对非幂等操作添加 retry。
- 只在应用层校验权限，数据层或服务层可绕过。
- 事务边界不覆盖所有写入，导致部分成功。

### Electron / 桌面端

重点关注：

- main/preload/renderer 边界是否清楚。
- IPC 参数、返回值、错误传播和权限边界。
- context isolation、nodeIntegration 和 sandbox。
- 跨平台路径、打包后资源路径、权限不足、自动更新。
- 子进程、监听器、文件句柄和退出清理。

常见坑：

- renderer 可传任意路径或命令给 main。
- 开发环境路径可用，打包后路径失效。
- 退出流程只关闭父进程，子进程残留。

### 文件系统 / 下载 / 安装

重点关注：

- 路径穿越、符号链接、allowlist、目标目录约束。
- 原子写入、临时目录、staging、校验、失败清理。
- hash/signature、内容类型、大小限制、断点续传。
- 权限不足、磁盘不足、并发下载和回滚。

常见坑：

- 解压未校验 entry path。
- 下载只检查 HTTP 200，不校验内容完整性。
- 直接覆盖目标文件，失败后无法恢复。

### 数据库 / Schema / 迁移

重点关注：

- expand/contract 顺序、旧版本兼容、双读/双写。
- 幂等迁移、备份、事务、锁风险、索引和性能。
- 大表数据量、回填节奏、失败重试和回滚。
- schema、API、UI、任务和报表的一致修改。

常见坑：

- 删除字段但旧版本或后台任务仍读取。
- 迁移不可重复运行。
- 长事务或全表锁影响线上。

### 异步 / 并发 / 任务调度

重点关注：

- 幂等、重入、取消、超时、重试退避。
- stale response、顺序保证、资源释放和部分失败。
- 队列可见性、死信、去重、任务恢复和状态机。

常见坑：

- 认为单线程就没有竞态。
- retry 包裹非幂等副作用。
- 任务成功写状态失败，或写状态成功但副作用失败。

### 安全 / 隐私 / 权限

重点关注：

- 输入校验、权限边界、注入、路径穿越。
- token、cookie、密钥、私有 payload 和用户隐私是否脱敏。
- 最小权限、默认拒绝、审计、滥用场景和负向测试。

常见坑：

- 信任客户端传入的 `userId`、`role`、`path`。
- 日志输出完整 token、cookie 或私有 payload。
- 为了调试临时放宽 CORS、权限或 sandbox。

### 性能 / 资源

重点关注：

- 热路径复杂度、重复 I/O、批量处理和缓存。
- 内存、CPU、文件句柄、连接、定时器和监听器清理。
- 懒加载、分页、虚拟列表、背压和限流。

常见坑：

- 小数据可行的 O(n^2) 放到生产数据失效。
- 缓存没有上限或 key 过于高基数。
- 每次请求重复初始化昂贵对象。

### AI Agent / 工作流 / Harness

重点关注：

- 阶段门：只读、方案、实现、验证、提交是否分离。
- 上下文：记忆、项目规则、diff、日志和任务边界是否保留。
- 工具权限：读写、网络、外部状态和确认门是否清楚。
- 多 agent：文件所有权、冲突、结果合并和验证责任。
- 可恢复性：中断、压缩、失败重试和最终证据。

常见坑：

- 一个超大 skill 覆盖所有流程，导致触发不准和上下文浪费。
- 多 agent 同时写同一文件或 lockfile。
- 没有证据就宣称完成。

## 决策矩阵细则

评分使用 1-5 分：

- 5：明显优秀，有直接项目证据或强行业依据。
- 4：较好，风险可控，有合理证据。
- 3：可接受，但存在明确 tradeoff。
- 2：弱适配，风险或成本偏高。
- 1：不建议，除非有特殊约束。

评估项解释：

- 需求匹配度：是否满足用户核心目标和验收标准。
- 项目适配度：是否复用现有架构、工具、模式和约束。
- 稳定性：是否降低核心链路、异常路径和边界场景风险。
- 流畅性：是否改善用户体验、开发体验或流程连续性。
- 安全性：是否符合最小权限、数据保护和输入边界。
- 实现工作量：上下文、编码、集成、测试、review 和发布总成本。
- 运维复杂度：日志、监控、告警、排障和依赖维护成本。
- 扩展性：是否支持合理演进，而不是过早平台化。
- 回滚简单度：失败后是否能低成本恢复。
- 验证信心：是否能用自动化或明确手工路径证明正确。

## AI 编程工作量估算

估算时使用以下结构：

```markdown
AI 工作量：M
- 上下文发现：M，原因：
- 方案细化：S，原因：
- 代码生成：S，原因：
- 集成修改：M，原因：
- 测试编写：M，原因：
- 验证执行：M，原因：
- Review 与加固：S，原因：
- 发布/回滚准备：XS，原因：
主要不确定性：
```

校准规则：

- 代码少不等于工作量低；公共契约、兼容性和验证会放大成本。
- AI 生成代码通常快，但上下文发现、集成、测试和 review 仍是主要成本。
- 触达核心链路、安全、权限、数据或发布，最低通常是 M。
- 需要迁移、跨端兼容、灰度或回滚，通常是 L 或 XL。
- 没有可运行验证时，工作量和风险都要上调。

## 方案反模式

避免以下输出：

- 只给一个方案，没有说明替代方案为何不适合。
- 方案很完整，但明显超过当前阶段需要。
- 方案很简单，但忽略核心链路、生产失败路径或安全边界。
- 只讲“业界最佳实践”，不结合当前项目代码和用户约束。
- 只讲优点，不讲缺点、适用场景和不适用场景。
- 没有证据来源，或把推断写成事实。
- 工作量只按“代码量”估算。
- 给实现 agent 的指导停留在口号，无法转成具体任务。

## 推荐方案校准

推荐方案必须满足：

- 当前阶段合适。
- 能覆盖用户核心目标。
- 风险与复杂度匹配。
- 能被项目现有结构吸收。
- 有清晰验证方式。
- 有明确边界和延后项。

推荐语句建议：

```markdown
我推荐方案 B。它比方案 A 多处理了生产中的失败路径和验证闭环，但没有引入方案 C 的长期平台化复杂度；按当前阶段，它能用可控工作量覆盖核心风险。建议暂缓 X 和 Y，除非后续出现 Z 需求。
```
