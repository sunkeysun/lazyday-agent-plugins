# 上下文恢复

## 目标

从目标仓库的 `.lazyday/coding/requirements/` 中恢复可继续执行的需求上下文，并根据用户当前描述选择最相关的候选需求。

本 skill 是只读的上下文感知和交接入口：

- 读取已有需求产物。
- 判断用户当前描述与哪些需求最相关。
- 给出候选排序、匹配依据、当前阶段和建议下一步 skill。
- 在上下文明确时，生成可直接交给后续 skill 的交接摘要。

本 skill 不负责创建、切换、归档、重命名或修复产物；这些动作交给 `manage-artifacts`。本 skill 也不负责方案、实现、验证、Review 或提交。

## 默认模式

默认只读。允许读取：

- `.lazyday/coding/requirements/index.md`
- `.lazyday/coding/requirements/CURRENT`
- 候选需求下的 `manifest.md`
- 候选需求下的 `handoff.md`
- 候选需求下的 `brief.md`
- 必要时读取少量阶段文件的标题、摘要或最近状态

禁止：

- 写入 `CURRENT`。
- 创建或修改需求目录。
- 修改业务代码。
- 自动把 `.lazyday/` 纳入 Git 提交。
- 为了匹配上下文加载所有阶段文件全文。

如果用户要求“切换到这个需求”“创建新需求”“修复产物”“整理 handoff”，输出建议后交给 `manage-artifacts` 执行。

## 读取流程

1. 确认当前目录在目标仓库内。
2. 读取适用的项目级规则。
3. 运行 `git status --short --branch`，只用于理解当前分支和工作区状态。
4. 检查 `.lazyday/coding/requirements/` 是否存在。
5. 读取 `index.md` 和 `CURRENT`。
6. 从 `index.md` 初筛候选需求，优先包括：
   - `CURRENT` 指向的需求。
   - 状态为 `active`、`in-progress`、`blocked` 或未完成的需求。
   - 标题、标签、涉及文件、阶段或更新时间与用户描述匹配的需求。
7. 对候选需求读取 `manifest.md`、`handoff.md`、`brief.md`。
8. 如仍无法判断，最多读取最相关的 3 个候选需求的关键阶段文件摘要。

如果没有任何产物目录，说明当前仓库没有可恢复的 Lazyday 需求产物，并建议用 `manage-artifacts` 创建新需求，或直接使用单一功能 skill。

## 匹配维度

按以下维度排序候选需求：

- **语义相关性**：用户当前描述是否匹配需求标题、目标、关键词、标签、问题现象或涉及文件。
- **当前指针**：`CURRENT` 指向的需求加权，但不要无条件采用。
- **更新时间**：最近更新的未完成需求优先。
- **阶段匹配**：用户要做的动作是否匹配该需求的下一阶段。
- **文件匹配**：当前 dirty worktree、用户提到的文件或模块是否出现在产物中。
- **状态匹配**：未完成、阻塞、待验证、待提交的需求优先于 archived/done。

不要只按最近更新时间选择；如果用户描述明显指向另一个需求，应优先语义匹配。

## 置信度策略

- **高置信**：只有一个明显候选，且用户描述、`CURRENT` 或文件范围一致。输出推荐需求和下一步，不写入 `CURRENT`。
- **中置信**：有 2-3 个合理候选。列出排序、匹配依据和差异，要求用户选择；如果用户允许，可建议交给 `manage-artifacts` 切换当前需求。
- **低置信**：候选弱相关或互相冲突。说明无法安全判断，建议用户补充需求关键词、文件、模块、阶段，或创建新需求。
- **无候选**：建议从 `clarify-task` 或 `manage-artifacts` 开始；用户只想单次执行时，也可以直接交给相应业务 skill。

## 阶段判断

根据产物文件和 `handoff.md` 判断下一步：

- 缺少任务契约或验收标准：`clarify-task`
- 缺少仓库上下文、入口、调用链或验证命令：`explore-repo`
- 缺少方案取舍或外部最佳实践判断：`research-approach`
- 方案已定但任务未拆：`break-down-task`
- 有 bug 现象但缺少根因证据：`diagnose-problem`
- 方案和边界已清楚且用户授权修改：`implement-change`
- 已实现但缺少证据：`verify-change`
- 已有 diff 需要审查：`review-code`
- 已验证且用户要求提交推送：`commit-changes`
- 需要创建、切换、归档或修复产物：`manage-artifacts`

## 交接规则

- 用户选择候选后，需要更新 `CURRENT`：交给 `manage-artifacts`。
- 用户要补任务契约：交给 `clarify-task`。
- 用户要继续探索、调研、拆解、诊断、实现、验证、Review 或提交：输出交接摘要后交给对应 skill。
- 用户只想单独执行某个 skill：不要强制创建或选择需求产物，只把可用上下文作为可选输入。
