# Review 评分标准

本文件用于 `review-code` 的深入评分、严重度校准和合入决策。只有在用户要求评分、完整报告、生产级把关，或改动风险达到 S2+ 时读取。

## 评分原则

评分是合入风险的压缩表达，不是主观印象。必须先有问题清单、测试证据和残余风险，再给分。

总分 100：

- 正确性 25：核心逻辑是否满足意图，边界和错误路径是否可靠。
- 回归安全 25：是否破坏已有行为、公共契约、兼容性、核心链路。
- 架构一致性 20：是否符合项目边界、抽象层级、依赖方向和已有模式。
- 测试充分性 15：是否覆盖新增行为、失败路径、边界、回归和关键集成。
- 可维护性 15：复杂度、命名、重复、可观测性、注释准确性、可演进性。

## 分数区间

- 90-100：可直接通过。没有 P0/P1；P2 很少且不影响合入；验证证据充分。
- 80-89：有条件通过。没有 P0；P1 不存在或非常轻微；P2 可在后续小修；测试缺口不影响核心路径。
- 65-79：需修改。存在 P1 或多个 P2；测试/回归/契约存在明显缺口。
- 40-64：高风险。多个 P1 或一个接近 P0 的问题；不建议合入。
- 0-39：拒绝。存在 P0，或无法判断改动是否安全。

评分限制：

- 有 P0 时，综合分不得高于 39，结论必须是“拒绝”。
- 有确定 P1 时，综合分通常不得高于 79，结论至少是“需修改”。
- 测试完全缺失且改动触达核心链路时，综合分通常不得高于 79。
- 只看到局部代码片段、无法确认调用链时，可以不给综合分，改为“信息不足，无法可靠评分”。

## 合入决策

- 通过：没有 P0/P1；P2 不影响当前合入；验证证据足够。
- 有条件通过：没有阻塞问题；存在低风险 P2/P3 或非核心测试缺口；明确条件后可合入。
- 需修改：存在 P1、多个 P2、关键测试缺口或架构不一致；修复后需要复审。
- 拒绝：存在 P0、不可接受的安全/数据/兼容风险，或 review 信息不足以判断核心链路安全。

## 证据等级

- A 级：直接从 diff、测试失败、类型错误、构建错误、官方契约或明确调用链得出。
- B 级：从现有代码模式、调用方搜索、配置路径和可推导状态流得出。
- C 级：基于经验的合理怀疑，但缺少完整上下文；只能作为“需要确认”或测试建议。

正式 P0/P1 问题项尽量使用 A/B 级证据。C 级证据不要写成确定结论。

## 领域专项校准

### 前端 / UI

重点检查：

- loading/error/empty 状态是否完整。
- 重复点击、请求竞态、stale response、卸载后 setState。
- 权限、路由、刷新、深链、移动端布局和可访问性。
- 组件职责是否混乱，是否重复造已有组件。

常见高风险：

- 提交按钮无禁用或幂等，导致重复创建/扣费/请求。
- 异步响应未按请求版本校验，旧响应覆盖新状态。
- 权限只在 UI 隐藏，没有后端或 API 层保护。

### 后端 / API

重点检查：

- 请求参数校验、权限、幂等、事务、分页、限流、错误码。
- 数据一致性、并发写、重试安全、超时和外部依赖降级。
- API 契约是否兼容旧调用方。

常见高风险：

- 修改返回结构但未更新所有调用方。
- 部分失败时已写入数据但没有补偿或事务。
- retry 包裹非幂等操作。

### Electron / 桌面端

重点检查：

- main/preload/renderer 边界和 context isolation。
- IPC 参数、返回类型、错误传播和权限边界。
- 跨平台路径、打包后路径、自动更新和退出清理。

常见高风险：

- renderer 可传任意路径给 main 执行文件操作。
- 开发环境路径可用，打包后资源路径失效。
- 子进程/监听器退出时未清理。

### 文件系统 / 下载 / 安装

重点检查：

- 路径穿越、原子写入、staging、hash/signature、失败清理。
- 磁盘不足、权限不足、断点重试、回滚。

常见高风险：

- 直接解压未校验 entry path。
- 覆盖目标文件前没有 staging 和回滚。
- 下载成功只检查 HTTP 200，不校验内容完整性。

### 数据库 / 迁移

重点检查：

- expand/contract 顺序、幂等、锁风险、旧版本兼容、备份和回滚。
- 索引、性能、批量数据、双写/双读窗口。

常见高风险：

- 删除字段但旧版本仍读取。
- 长事务或全表锁影响线上。
- 迁移不可重复运行。

### 安全 / 隐私

重点检查：

- 输入校验、权限边界、注入、敏感日志、token/secret 暴露。
- 最小权限、审计事件、默认安全策略。

常见高风险：

- 日志记录完整 token、cookie、私有 payload。
- 服务端信任客户端传入的 userId、role、path。
- 放宽 CORS、权限或 sandbox 而无负向测试。

### 性能 / 资源

重点检查：

- 热路径复杂度、重复 I/O、批量处理、缓存、懒加载。
- 内存、CPU、文件句柄、连接、定时器、监听器清理。

常见高风险：

- 在渲染循环或请求热路径做 O(n^2) 扫描。
- 每次请求重复初始化昂贵对象。
- 缓存无上限或 key 包含高基数字段。

## 反模式

Review 时避免：

- 只复述代码做了什么，不判断风险。
- 因为想给意见而输出无影响的风格建议。
- 没有证据就说“可能有问题”。
- 只说“补测试”，不说补哪个路径。
- 看到新增代码就默认低风险，忽略它是否接入核心链路。
- 看到 try/catch 就默认安全，忽略吞错、状态不一致和观测缺失。
- 评分和问题清单不一致。
- 把作者需要解释的问题写成确定 bug。
