# 变更验证

## 目标

为当前改动或指定修复建立可信验证证据。验证要覆盖用户要求、原失败路径、关键回归路径和已知风险，不用宽泛命令掩盖真正缺口。

## 默认模式

默认只读验证。

允许：

- 读取代码、diff、测试、配置、日志和项目文档。
- 运行已有测试、类型检查、lint、构建、smoke 或只读诊断命令。
- 输出验证矩阵、通过证据、失败证据和剩余风险。

禁止：

- 未经明确要求修改文件、更新 snapshot、放宽测试或隐藏失败。
- 安装依赖、清理工作区、切换分支、提交代码或执行破坏性命令。
- 把未运行的检查写成已验证。

如果用户说“继续验证”“验证上次改动”但需求或改动上下文不明确，先使用 `resume-context` 匹配候选需求和可用产物。

## 产物交接

需求产物由 `manage-artifacts` 统一管理。需要创建、选择、切换、列出、归档或修复 `.lazyday/coding/requirements/` 时，先使用 `manage-artifacts`。如果用户说“继续”“上次”“当前需求”但没有明确需求 ID，先使用 `resume-context` 匹配候选需求。

当前需求已明确时，本 skill 只读取和更新与变更验证相关的产物。

本 skill 优先读取：

- `brief.md`：验收标准和用户目标。
- `repo-context.md`：测试入口、验证命令、核心链路和风险边界。
- `research.md`：方案中的验证策略。
- `plan.md`：任务验证矩阵和每个任务完成标准。
- `diagnosis.md`：原失败路径、复现方式和预期修复证据。
- `implementation.md`：实际改动范围和未验证风险。
- `handoff.md`：上一阶段最短交接摘要。

当用户要求跨会话记录、继续交接或允许落盘，且当前需求已明确时，把验证结果写入 `verification.md`，并同步更新 `handoff.md`。不要默认提交 `.lazyday/`。

## 执行流程

1. 明确验证对象：当前 diff、指定文件、某个修复、某条失败路径或交付清单。
2. 读取适用项目规则和工作区状态。
3. 用 `git diff --stat`、`git diff --name-status` 和必要的局部 diff 建立变更地图。
4. 从变更地图推导最小验证集：测试、类型检查、lint、构建、smoke、日志或手工路径。
5. 优先验证原失败路径；其次验证受影响调用方和关键边界。
6. 运行验证并保留命令、结果和关键输出。
7. 如果验证失败，先分析失败是否来自本次改动、环境问题、既有失败或测试本身问题。

## 验证矩阵

输出时按需列出：

- 需求覆盖：每个用户目标是否有验证证据。
- 回归覆盖：相关调用方、契约、配置和边界是否检查。
- 自动化证据：命令、结果、关键输出。
- 手工证据：步骤、预期、实际结果。
- 未覆盖项：原因、风险、建议下一步。

安全敏感路径必须把已知攻击/绕过样例写进验证矩阵。涉及 CSV、spreadsheet 或导出文件时，危险前缀至少覆盖 `=`, `+`, `-`, `@`, tab、CR、LF，以及前导空白或控制字符绕过；如果项目已有更严格安全规范，以项目规范为准。

## 失败处理

验证失败时不要直接修。先说明：

- 失败命令。
- 失败摘要。
- 初步归因。
- 是否阻塞交付。
- 最小修复或取证建议。

用户明确要求继续修复时，再交给 `implement-change` 或按当前任务授权进入实现。

## 输出

默认输出：

```markdown
**验证结果**
- 结论：
- 已运行：
- 通过证据：
- 失败或跳过：

**覆盖范围**
- 已覆盖：
- 未覆盖：
- 剩余风险：

**下一步**
- 
```

单独使用本 skill 时，最后明确本轮是否已完成：如果用户只要求验证，输出通过/失败证据和剩余风险即完成；验证失败时只给定位或修复建议，不自动改代码。
